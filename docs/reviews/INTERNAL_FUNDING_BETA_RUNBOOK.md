# Internal Funding Beta Runbook

Date: 2026-06-19

## Purpose

This runbook describes how the owner should operate POLY during Controlled Internal Funding Beta. It is preparation only; it does not deploy production.

## Operating Boundary

Allowed:

- allowlisted internal users only.
- supported token on supported chain only.
- self-managed EVM deposit addresses generated by the platform.
- encrypted private keys stored server-side only.
- deposit monitor auto-credit after confirmations when owner enables it.
- withdrawal requests with ledger hold.
- admin manual payout review.
- admin manual payout transaction outside the app.
- admin records payout tx hash.

Not allowed:

- public funding.
- anonymous funding.
- automatic withdrawal broadcast.
- live bot trading.
- production launch.
- removing allowlist.
- bypassing kill switch.

## Pre-Start Checklist

1. Confirm target commit is on `dev`.
2. Confirm PR #220 is resolved if wallet/deposit UI is needed.
3. Confirm no unexpected dirty files on server.
4. Configure env vars privately. Do not print values.
5. Run Prisma generate and validate.
6. Run TypeScript validation.
7. Run test suite.
8. Confirm funding kill switch starts as true.
9. Confirm bot live trading flags remain disabled.
10. Confirm admin account access.

## Start Funding Beta

Recommended staged start:

1. Start app with funding disabled:
   - `INTERNAL_FUNDING_BETA_ENABLED=false`
   - `FUNDING_KILL_SWITCH=true`
   - `ALLOW_AUTO_DEPOSIT_CREDIT=false`
2. Verify public pages load.
3. Verify anonymous funding access is blocked.
4. Enable internal funding beta for allowlisted users:
   - `INTERNAL_FUNDING_BETA_ENABLED=true`
   - keep `FUNDING_KILL_SWITCH=true`
   - keep `ALLOW_AUTO_DEPOSIT_CREDIT=false`
5. Verify non-allowlisted user is blocked.
6. Verify allowlisted user sees funding blocked by kill switch.
7. Turn kill switch off only for controlled test window:
   - `FUNDING_KILL_SWITCH=false`
8. Verify allowlisted user can obtain deposit address.
9. Enable auto-credit only when owner is ready:
   - `ALLOW_AUTO_DEPOSIT_CREDIT=true`

## Deposit Drill

1. Use one allowlisted internal tester.
2. Generate or view assigned deposit address.
3. Send only supported token on supported chain.
4. Wait required confirmations.
5. Run or wait for deposit monitor.
6. Verify deposit row status.
7. Verify ledger deposit credit.
8. Verify balance increased once.
9. Run scan again and verify no duplicate credit.

## Withdrawal Drill

1. Allowlisted tester submits withdrawal request.
2. Verify request status is pending review.
3. Verify funds are held/locked.
4. Admin reviews request.
5. For rejection drill:
   - reject request.
   - verify held funds are released.
6. For completion drill:
   - admin manually sends payout outside the app.
   - admin records payout tx hash.
   - verify request completed.
   - verify no automatic app broadcast occurred.

## Emergency Stop

Set:

- `FUNDING_KILL_SWITCH=true`
- `ALLOW_AUTO_DEPOSIT_CREDIT=false`

Then restart/reload app services according to the server process manager.

Effects:

- blocks deposit address get/create.
- blocks withdrawal request creation.
- blocks deposit monitor auto-credit entrypoints.

## Bot Safety

Keep:

- `POLY_BOTS_ENABLED=false`
- `POLY_BOTS_LIVE_TRADING=false`
- `POLY_BOTS_GLOBAL_KILL_SWITCH=true`
- `POLY_BOTS_MODE=dryRun`
- `LIVE_SYSTEM_LIQUIDITY_ENABLED=false`
- `SYSTEM_LIQUIDITY_DRY_RUN=true`

Do not start live bot services during funding beta.

## Operator Notes

- Never print `.env` values.
- Never copy private keys into tickets, docs, logs, or chat.
- Never expose raw or encrypted private keys to frontend.
- Use small test amounts first.
- Keep the allowlist small.
- Stop immediately if duplicate credit, unexpected balance change, or secret leakage is suspected.
